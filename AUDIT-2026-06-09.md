## TL;DR verdict

This is a strong codebase. The part that's hardest to get right (a deterministic,
fixed-point, replayable economy engine) is genuinely well built and well tested. I found
**no correctness bug in the engine math, the §7 invariant, or the determinism contract.**
Those are enforced in code, not just asserted in comments.

The real gaps are about completeness and process, not broken logic: there's **no save
system yet**, **CI doesn't test the app**, content lives in **three hand-synced copies**,
and there's a **large uncommitted working tree on a stale branch name**. Nothing here is a
fire. Most of it is the normal "we're mid-build" punch list, and the author already logs
most of it in STATE.md.

Severity counts: Critical 0, High 2, Medium 4, Low 7.

## What's verified solid (don't let anyone "fix" these)

- **Fixed-point integer discipline holds.** No `double` type anywhere in `packages/engine`
  (grep-verified; the word only appears in comments). `purity_guard_test.dart` scans every
  lib file for `double` / `.toDouble(` / `dart:math` / `DateTime` / `Random(`, proves the
  scan isn't vacuous, and even self-tests that the detector bites planted violations and
  ignores `doubled` / `SplitMix64Rng(`. That's the opposite of test theater.
- **The §7 invariant is real and enforced at runtime.** `netWorthCents` is a getter with no
  field and no setter (`model.dart:758`), so nothing can write net worth. Card deltas are
  rejected at apply time if any key falls outside `{ebitda, multiple, netDebt, own, cash}`
  (`apply.dart:814`), and `invariant_test.dart` re-checks all 33 cards against the same set.
- **Determinism is contract-grade.** SplitMix64 with an explicit cursor, no clock or ambient
  RNG. `golden_replay_test.dart` pins a full scripted run byte-for-byte and, crucially,
  hand-verifies the 28-draw count with a documented per-draw breakdown (not just "diff against
  a generated file"). Retired goldens v1..v6 are sealed with proofs that old streams diverge.
- **The one-way dependency holds.** `packages/engine` depends only on `json_annotation`; the
  app depends on the engine via `path: ../packages/engine`. No Flutter import leaks into the
  engine.
- **The app mirrors the discipline.** `GameController` holds engine state and dispatches; it
  computes nothing economic, and every UI-side number is either an engine value or a clearly
  documented display mirror. Animation controllers are disposed. The controller is disposed
  in `main.dart`.
- **Money math defers divisions correctly.** EV, equity, and ownership all truncate last
  (`resolver.dart`, `model.dart:761-764`), matching the locked convention.

The content copies (`data/`, `app/assets/data/`, `packages/engine/assets/`) are currently
byte-identical (md5-verified for cards.json; jq-diff clean for both files).

## Findings by severity

### HIGH

**H1 — No save / persistence layer.** The game can't save or resume. Close the app mid-run
and the run is gone; the title screen has no CONTINUE. `main.dart:78` literally comments
"CONTINUE/persistence is Phase 4," and there's no `shared_preferences`, `path_provider`,
`dart:io`, or file write anywhere in `app/`. The save format spec (`docs/06`) exists but is
unimplemented. For a roguelike this is the single biggest functional gap. The engine is
already replay-serializable (`flatten()` + schemaVersion), so the foundation is there. This
is the next planned phase, so it's deferred-by-design, not an oversight.

**H2 — CI tests the engine only; the app has zero CI coverage.** `.github/workflows/ci.yml`
runs entirely inside `packages/engine` (pub get, build_runner, `dart test`, `dart analyze`,
a sim smoke). The 33 Flutter widget tests in `app/test/` never run in CI, and there's no
`flutter analyze` or build gate. App regressions can land green. Two smaller issues ride
along: CI uses `sdk: stable` (floating) while the project pins Dart 3.12.1 locally, so the
analyzer can shift under you, and the workflow triggers `on: push` only with no
pull-request gate.

### MEDIUM

**M1 — Three hand-synced copies of the content JSON.** `cards.json` and
`economy-model.json` each exist in `data/`, `app/assets/data/`, and
`packages/engine/assets/`. They're identical today, but nothing enforces that. No test diffs
them, and there's no build step that generates the copies from one source. The engine tests
read `data/cards.json` while the app loads `app/assets/data/`, so a future edit to one copy
is silent drift between what's tested and what ships. Fix: a one-line sync test, or generate
the asset copies from `data/` at build.

**M2 — Large uncommitted working tree on a stale branch.** 29 modified files (engine `lib/`
and `test/`, plus android config) sit uncommitted on branch `feat/phase-0-toolchain`, while
the project is at Phase 5. The branch name no longer means anything, `main` is behind, and
it's unclear what's work-in-progress versus landed. Risk of lost or confused work. Commit,
rename to a phase-appropriate branch, and fast-forward main.

**M3 — Finite 64-bit head-room with an unbounded endless mode.** Net worth multiplies
`ebitda * multipleMilli` and `equity * ownershipBp` before truncating. In a normal bounded
run these stay far inside int64. But T5 "endless" never terminates (see L1) and organic
growth compounds at 20%/round, so the products are theoretically unbounded over a very long
endless run. There's no saturating clamp. Low probability, but worth a documented endless
cap or a saturating guard so a marathon run can't silently wrap.

**M4 — The engine trusts upstream magnitudes by design.** `apply()` takes raw action
payloads and doesn't validate signs or magnitudes; a negative cost would invert an action's
economics. The content layer is supposed to guarantee values, and `content_lint_test.dart`
does enforce a lot. The author logs this trust boundary in STATE.md's audit notes. It's fine
today; it gets fragile if content authoring widens without the lint widening with it.

### LOW

**L1 — T5 endless is a stub.** `DEADLINE_CHECK` always advances in T5, there's no escalation
or termination, and `controller.enterEndless()` is documented as the minimal sanctioned
entry. Endless never actually ends.

**L2 — Placeholder "UI dials."** The reroll fee is a flat $15k (`controller.dart:614`); the
doc 02 §4 scaling formula isn't implemented. Reinvest amount is a UI percent. Held plays with
per-venture deltas always auto-target the platform with no aim step. All three are documented
as future work.

**L3 — Deferred engine hooks (author-logged).** `PLY_SECONDARY_SALE` proceeds are a $0
placeholder; `ScheduledCost` is a minimal slice with no EARN_OUT bases or countdown;
`TakeDebt`'s cold-market gate needs a content flag; the partner fixed-cost face isn't in the
v1 card schema yet. All wired but not fully fleshed.

**L4 — Minor comment/doc drift.** `resolver.dart`'s F4 header still says "interestMax = 0.14
(800..1200 bp)" after the R12 tune moved it to 0.12/1200 (the constant and code are correct;
only the prose is stale). `golden_replay_test.dart`'s `replay()` docstring still says
"scripted v5 run" though it pins v7. Cosmetic only.

**L5 — `main()` has no error handling around content load.** A malformed bundled JSON would
throw out of `rootBundle.loadString` / `loadCards` and crash to a blank screen with no
user-facing error. Assets are bundled so this is unlikely in production, but a try/catch with
an error screen is cheap insurance.

**L6 — iOS not started.** Android-first is the stated plan, and there's no iOS build or
config yet. Expected, but it's "missing" against the stated iOS + Android target.

**L7 — Operational: a stale `.git/index.lock` was present** (likely from an interrupted git
or tool run) and would block commits. I removed it during this audit. Flagging in case it
recurs.

## Plain-English: what is this build?

**What it is.** A single-player, offline mobile game where you play a private-equity
investor. You run companies ("ventures"), grow their earnings, use debt and equity, buy bolt-on
acquisitions, and time exits, all to grow your net worth past rising tier bars before a
round deadline. The signature move is "multiple arbitrage": buy a small company cheap and
fold it into a bigger one that the market values at a higher multiple, and the gap becomes
instant value. It's a finance-themed roguelike: each run is a fresh randomized attempt, and
the same seed always replays identically.

**What works right now.** The whole rules engine is done and trustworthy. You can start a
run, operate companies each round (market weather shifts, interest gets charged, earnings
drift, neglected companies decay), play a hand of cards, hire partners and CEOs, take debt,
raise equity, do the arbitrage merge, take dividend recaps, and exit companies, all the way
to winning at the $1B tier or dying by bankruptcy or a missed deadline. There's a real UI for
all of it: title screen, the main run screen, the per-round digest, the deal "napkin" preview,
the arbitrage and exit flash beats, the deadline panel, and win/lose screens. The economy has
been balance-tuned with a Monte-Carlo harness, not just guessed at. It builds and runs on
Android.

**The drawbacks / rough edges.** A few mechanics are placeholders dressed up as final: the
reroll fee is a flat number instead of the real scaling formula, "endless mode" after you win
doesn't really escalate or end, and some cards (secondary sale, earn-outs) aren't fully wired.
A couple of in-game numbers auto-target the obvious choice instead of letting you aim.

**What's missing.** Three big ones. First and most important: **you can't save.** There's no
save or resume, so a run only exists while the app is open. Second: **iOS isn't started** yet
(Android only for now). Third: **the app side isn't protected by CI**, so UI breakage can slip
through even though the engine is well guarded. Smaller missing pieces: nicer "juice" polish,
proper company display names, and an unlimited-mode ending.

**One-line summary.** The brain of the game is excellent and battle-tested; the body is a
working Android playable that still needs a save system, iOS, and CI on the UI before it's
shippable.

## Recommended next steps (priority order)

1. **Build the save layer (H1).** It's the biggest unlock and the engine already supports it
   via `flatten()` + schemaVersion. Wire load-time schema-mismatch discard per docs/06.
2. **Add the app to CI (H2):** `flutter analyze` + `flutter test`, and pin the Dart/Flutter
   version instead of floating `stable`. Add a `pull_request` trigger.
3. **Kill the content-copy drift risk (M1):** one sync test that diffs the three copies, or
   generate the asset copies from `data/` at build.
4. **Commit and re-branch (M2):** land the 29-file working tree, rename off
   `feat/phase-0-toolchain`, fast-forward `main`.
5. **Decide endless mode (L1/M3):** either give T5 a real termination/escalation or cap it,
   and add a saturating guard on the net-worth products while you're there.
6. Sweep the stale comments (L4) and add the `main()` error screen (L5) as cheap polish.
